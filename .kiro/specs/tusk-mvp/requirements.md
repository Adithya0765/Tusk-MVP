# Requirements Document

## Introduction

TUSK is an AI debate platform MVP where users submit a topic and two AI models — Gemini 2.0 Flash (FOR) and Grok 3 Mini (AGAINST) — debate it across multiple rounds, producing a structured transcript and conclusion document. The platform is monetised via three subscription tiers (Free, Starter, Pro) billed in INR, with authentication via Clerk, data persistence in Supabase, payments via Stripe, and deployment on Vercel. The existing Next.js 14 codebase with its Tailwind design system, TUSK branding, and UI components serves as the visual foundation.

Claude (Anthropic) is supported as a future upgrade path for Agent A — it activates automatically when `CLAUDE_ENABLED=true` and `ANTHROPIC_API_KEY` are set, requiring no code changes.

---

## Glossary

- **Platform**: The TUSK Next.js 14 web application.
- **User**: An authenticated person who has signed up via Clerk.
- **Guest**: An unauthenticated visitor browsing the Platform.
- **Debate_Session**: A single debate job consisting of ordered turns between Agent_A and Agent_B, followed by a Conclusion.
- **Agent_A**: Gemini 2.0 Flash (or Claude when `CLAUDE_ENABLED=true`) assigned the FOR position in a Debate_Session.
- **Agent_B**: Grok 3 Mini assigned the AGAINST position in a Debate_Session.
- **Turn**: A single response from Agent_A or Agent_B within a Debate_Session.
- **Round**: One paired exchange — one Turn from Agent_A followed by one Turn from Agent_B.
- **Conclusion**: A structured summary document generated by Agent_A after all Rounds complete.
- **Debate_Engine**: The server-side module (`lib/debate-engine.ts`) that orchestrates Rounds and generates the Conclusion.
- **AI_Provider**: The abstraction layer (`lib/ai-provider.ts`) that routes calls to Gemini, Grok, or Claude.
- **Key_Manager**: The module (`lib/key-manager.ts`) that rotates API keys and enforces per-minute/per-day rate limits.
- **Subscription_Tier**: One of three plan levels — Free, Starter, or Pro — that governs debate quota and Round count.
- **Quota**: The number of Debate_Sessions a User may start within a calendar month under their Subscription_Tier.
- **Dashboard**: The authenticated page at `/dashboard` showing a User's Debate_Sessions and Quota usage.
- **Shared_View**: The public, unauthenticated page at `/share/[id]` displaying a completed Debate_Session.
- **Webhook**: An HTTP callback from Clerk or Stripe used to synchronise external events with the Platform's database.
- **Notification**: An optional email sent via Resend when a Debate_Session completes or fails.

## Requirements

### Requirement 1: User Authentication

**User Story:** As a Guest, I want to sign up and sign in using Clerk, so that I can access the Platform's debate features securely.

#### Acceptance Criteria

1. WHEN a Guest navigates to `/sign-up` or `/sign-in`, THE Platform SHALL render the Clerk-hosted authentication component.
2. WHEN a User successfully authenticates, THE Platform SHALL redirect the User to `/dashboard`.
3. WHEN a Clerk `user.created` webhook event is received, THE Platform SHALL create a corresponding record in the `users` table in Supabase within 5 seconds.
4. IF a Clerk webhook payload fails signature verification, THEN THE Platform SHALL return HTTP 400 and discard the payload.
5. WHILE a User session is active, THE Platform SHALL include the Clerk session token in all authenticated API requests.

---

### Requirement 2: Dashboard

**User Story:** As a User, I want a dashboard that shows my Debate_Sessions and Quota usage, so that I can track my activity and know how many debates I have remaining.

#### Acceptance Criteria

1. WHEN a User navigates to `/dashboard`, THE Platform SHALL display the User's current Quota usage as a progress bar showing debates used out of the monthly limit.
2. WHEN a User has no Debate_Sessions, THE Platform SHALL display an empty-state prompt with a call-to-action to start a new debate.
3. WHEN a User has one or more Debate_Sessions, THE Platform SHALL display each session as a card showing topic, status (Processing / Complete / Failed), and creation date.
4. WHEN a User clicks a session card, THE Platform SHALL navigate the User to `/debate/[id]`.
5. THE Platform SHALL refresh Quota usage data on each Dashboard page load without requiring a full browser reload.

---

### Requirement 3: New Debate Submission

**User Story:** As a User, I want to submit a debate topic, so that the Platform can start a Debate_Session between Agent_A and Agent_B.

#### Acceptance Criteria

1. WHEN a User navigates to `/debate/new`, THE Platform SHALL display a textarea accepting up to 500 characters and a submit button.
2. THE Platform SHALL display a live character count as the User types in the topic textarea.
3. IF a User submits a topic with 0 characters or more than 500 characters, THEN THE Platform SHALL display a validation error and prevent submission.
4. IF a User's monthly Quota is exhausted, THEN THE Platform SHALL display an upgrade prompt and prevent submission.
5. IF the number of Debate_Sessions with status `processing` for the User equals or exceeds 3, THEN THE Platform SHALL display a concurrency error and prevent submission.
6. WHEN a valid topic is submitted, THE Platform SHALL create a Debate_Session record with status `processing` and redirect the User to `/debate/[id]`.
7. WHEN a valid topic is submitted, THE Platform SHALL invoke the Debate_Engine as a Vercel serverless function with `maxDuration=60`.

---

### Requirement 4: Debate Engine Execution

**User Story:** As a User, I want the Platform to run a structured debate between two AI models, so that I receive a balanced, multi-round exchange on my topic.

#### Acceptance Criteria

1. WHEN a Debate_Session starts, THE Debate_Engine SHALL assign Agent_A (Gemini / Claude) the FOR position and Agent_B (Grok) the AGAINST position. Agent labels displayed to the user SHALL reflect the active provider via `getAgentNames()`.
2. WHEN a Debate_Session starts, THE Debate_Engine SHALL execute the number of Rounds corresponding to the User's Subscription_Tier: 2 Rounds for Free, 3 Rounds for Starter, 5 Rounds for Pro.
3. THE Debate_Engine SHALL limit each Turn to a maximum of 300 tokens.
4. THE Debate_Engine SHALL execute a maximum of 12 Turns per Debate_Session.
5. WHEN all Rounds complete, THE Debate_Engine SHALL invoke Agent_A to generate the Conclusion.
6. WHEN the Conclusion is generated, THE Debate_Engine SHALL persist the Conclusion to the `conclusions` table and update the Debate_Session status to `complete`.
7. IF any AI API call returns an error after 3 retry attempts, THEN THE Debate_Engine SHALL update the Debate_Session status to `failed` and record the error.
8. WHEN a Debate_Session status changes to `complete` or `failed`, THE Debate_Engine SHALL decrement the User's available Quota by 1.

---

### Requirement 5: Debate View

**User Story:** As a User, I want to view the progress and results of a Debate_Session, so that I can read the transcript and download the conclusion.

#### Acceptance Criteria

1. WHEN a User navigates to `/debate/[id]` and the session status is `processing`, THE Platform SHALL display a loading indicator and poll the `/api/debate/[id]/status` endpoint every 5 seconds.
2. WHEN a User navigates to `/debate/[id]` and the session status is `complete`, THE Platform SHALL display the full Turn-by-Turn transcript and the Conclusion document.
3. WHEN a User navigates to `/debate/[id]` and the session status is `failed`, THE Platform SHALL display an error message and a retry option.
4. WHEN a User clicks the download button on a completed Debate_Session, THE Platform SHALL generate and download a PDF or Markdown file containing the transcript and Conclusion.
5. WHEN a User clicks the share button on a completed Debate_Session, THE Platform SHALL copy the `/share/[id]` URL to the clipboard and display a confirmation toast.
6. IF a User attempts to navigate to `/debate/[id]` for a session that does not belong to them, THEN THE Platform SHALL return a 404 page.

---

### Requirement 6: Shared Debate View

**User Story:** As a User, I want to share a completed Debate_Session via a public URL, so that anyone can read the debate without signing in.

#### Acceptance Criteria

1. WHEN a Guest navigates to `/share/[id]`, THE Platform SHALL display the topic, Turn-by-Turn transcript, and Conclusion without requiring authentication.
2. IF a Guest navigates to `/share/[id]` for a session with status other than `complete`, THEN THE Platform SHALL display a message indicating the debate is not yet available.
3. IF a Guest navigates to `/share/[id]` for a non-existent session ID, THEN THE Platform SHALL return a 404 page.
4. THE Platform SHALL render the Shared_View with Open Graph metadata including the debate topic as the page title, enabling link previews on social platforms.

---

### Requirement 7: Subscription and Payments

**User Story:** As a User, I want to subscribe to a paid plan, so that I can access more debates and more Rounds per debate.

#### Acceptance Criteria

1. WHEN a Guest navigates to `/pricing`, THE Platform SHALL display the three Subscription_Tiers — Free (₹0, 3 debates/month, 2 Rounds), Starter (₹299/month, 20 debates/month, 3 Rounds), and Pro (₹799/month, 60 debates/month, 5 Rounds).
2. WHEN an authenticated User clicks an upgrade button on `/pricing`, THE Platform SHALL initiate a Stripe Checkout session and redirect the User to the Stripe-hosted payment page.
3. WHEN a Stripe `checkout.session.completed` webhook event is received, THE Platform SHALL update the User's Subscription_Tier and reset the monthly Quota within 10 seconds.
4. WHEN a Stripe `customer.subscription.deleted` webhook event is received, THE Platform SHALL downgrade the User's Subscription_Tier to Free and adjust the Quota accordingly.
5. IF a Stripe webhook payload fails signature verification, THEN THE Platform SHALL return HTTP 400 and discard the payload.
6. THE Platform SHALL reset each User's monthly Quota to the limit of their current Subscription_Tier on the first day of each calendar month.

---

### Requirement 8: Settings

**User Story:** As a User, I want to manage my account settings, so that I can update my display name and control email notifications.

#### Acceptance Criteria

1. WHEN a User navigates to `/settings`, THE Platform SHALL display the User's current display name in an editable text field.
2. WHEN a User submits a new display name, THE Platform SHALL persist the change to the `users` table and display a success confirmation.
3. WHEN a User navigates to `/settings`, THE Platform SHALL display a toggle for email notifications showing the current preference.
4. WHEN a User toggles the email notification preference, THE Platform SHALL persist the change to the `users` table immediately without requiring a separate save action.

---

### Requirement 9: Email Notifications

**User Story:** As a User, I want to receive an email when my debate completes or fails, so that I can return to the Platform to review the results.

#### Acceptance Criteria

1. WHERE email notifications are enabled for a User, WHEN a Debate_Session status changes to `complete`, THE Platform SHALL send a notification email via Resend containing a link to `/debate/[id]` within 60 seconds of completion.
2. WHERE email notifications are enabled for a User, WHEN a Debate_Session status changes to `failed`, THE Platform SHALL send a notification email via Resend containing a link to `/debate/[id]` within 60 seconds of failure.
3. THE Platform SHALL log each notification attempt to the `notifications_log` table, recording the Debate_Session ID, recipient email, status (sent / failed), and timestamp.
4. IF a Resend API call fails, THEN THE Platform SHALL log the failure to the `notifications_log` table and not retry automatically.

---

### Requirement 10: Landing Page

**User Story:** As a Guest, I want to view a compelling landing page, so that I can understand what TUSK does and decide to sign up.

#### Acceptance Criteria

1. WHEN a Guest navigates to `/`, THE Platform SHALL display a hero section, a "How It Works" section, a features section, a pricing summary, and a footer — all reusing the existing Tailwind design system and TUSK branding.
2. WHEN a Guest clicks the primary call-to-action on the landing page, THE Platform SHALL navigate the Guest to `/sign-up`.
3. THE Platform SHALL render the landing page with a Lighthouse performance score of 80 or above on desktop.
4. THE Platform SHALL render the landing page with correct Open Graph metadata for social sharing, including title, description, and image.

---

### Requirement 11: API Rate Limiting and Cost Controls

**User Story:** As the Platform operator, I want hard limits on AI token usage and concurrent jobs, so that infrastructure costs remain predictable.

#### Acceptance Criteria

1. THE Debate_Engine SHALL enforce a maximum of 300 tokens per Turn for all Agent_A and Agent_B responses.
2. THE Debate_Engine SHALL execute a maximum of 12 Turns per Debate_Session regardless of Subscription_Tier.
3. THE Platform SHALL enforce a maximum of 3 concurrent Debate_Sessions per User at any point in time.
4. THE Platform SHALL enforce a maximum topic length of 500 characters at both the client and server layers.
5. IF any enforced limit is exceeded at the server layer, THEN THE Platform SHALL return HTTP 422 with a descriptive error message.

---

### Requirement 12: AI Key Management

**User Story:** As the Platform operator, I want API keys to be rotated automatically across multiple keys per provider, so that free-tier rate limits are maximised without manual intervention.

#### Acceptance Criteria

1. THE Key_Manager SHALL support up to 5 API keys per provider (Gemini and Grok).
2. THE Key_Manager SHALL track requests-per-minute and requests-per-day per key, enforcing Gemini limits (14 RPM / 1400 RPD) and Grok limits (25 RPM / 500 RPD).
3. WHEN a key receives a 429 response, THE Key_Manager SHALL block that key for 65 seconds before retrying it.
4. WHEN all keys for a provider are rate-limited, THE Platform SHALL return an error to the caller indicating temporary unavailability.
5. THE AI_Provider module SHALL route Agent_A calls to Claude when `CLAUDE_ENABLED=true` and `ANTHROPIC_API_KEY` is set, falling back to Gemini otherwise — with no code changes required.
